17 /// A class for managing application configuration and execution
18 /** Derived classes should implement at a minimum
19  *
20  * \code
21  virtual void setupConfig();
22  virtual void loadConfig();
23  virtual int execute();
24  \endcode
25  *
26  * These are executed in the order shown by the call to \ref main().
27  *
28  * This class uses a cascaded configuration system. The application configuration is built up from the following sources, in increasing order of precedence:
29  * - A global configuration file
30  * - A user configuration file
31  * - A local configuration file
32  * - A configuration file specified on the command line
33  * - The command line
34  *
35  * At each step in the above order, values read from that configuration source override any previous settings. So the command line has the highest precedence.
36  *
37  * The configuration files are ini-style, with sections. That is
38  \verbatim
39  key1=value1
40  key2=value2
41 
42  [section1]
43  key3=value3
44  key4=value4,value4.1, value4.2, value4.3
45 
46  [section2]
47  key3=value5
48  key3=value5.1
49  key4=value6_over_
50  multiple_lines
51 
52  \endverbatim
53  * such that section1.key3 is distinct from section2.key3 (they must have different config-target names though). Additional entries within one file with the same
54  * section and key are appended with a newline to the previous entry. So the value of section2.key3 is "value5\\nvalue5.1". Multi-line values are handled such
55  * that in the above example the result is key4=value6_over_multiple_lines. Vectors are input as in section1.key4.
56  *
57  * \todo add handling of +=
58  *
59  * The command line parser handles both short-opt ("-h -vArg -n Arg") and long-opt ("--help --value=Arg --number=Arg") styles.
60  *
61  * \note After loadConfig() but before execute(), the containers in \ref config are de-allocated , so they can not be used inside execute.
62  *
63  * \ingroup mxApp
64  */
